# 🛠 HydraLLM 트러블슈팅 가이드

[English](TROUBLESHOOTING.md) | [한국어](TROUBLESHOOTING.ko.md)

이 문서는 로그를 통해 문제를 진단하고 해결한 사례와 패턴을 기록하여, 향후 AI 에이전트와 개발자가 문제를 더 쉽게 해결할 수 있도록 돕습니다.

---

## 🔍 로그 기반 진단 워크플로우

"문제 발생" 보고를 받았을 때의 조치 순서:
1. **ID 식별**: 사용자가 제공한 세션 ID로 로그를 검색(grep)하여 연관된 `[req_...]` 패턴의 Request ID를 찾습니다.
2. **생애주기 추적**: 해당 Request ID를 공유하는 모든 로그를 모아 전체 흐름을 파악합니다:
   - 분석(Analysis) -> 웹 페칭(Web Fetch) -> 라우팅(Routing) -> 제공자 선택 -> 모델 응답.
3. **Usage 필드 확인**: 응답 메타데이터의 `gateway_provider`와 `gateway_key_index`를 확인하여 정확히 어떤 자원에서 오류가 났는지 특정합니다.

---

## ✅ 해결된 문제 및 패턴

### 1. TypeError: cannot unpack non-iterable NoneType object
- **증상**: 500 내부 서버 오류 발생. `gateway.py`의 `_process_with_retries` 결과 언패킹 시 `TypeError` 발생.
- **원인**: 리트라이 루프가 값을 반환하거나 예외를 던지지 않고 종료됨 (마지막 시도의 폴백 실패 등). 이 경우 파이썬은 암시적으로 `None`을 반환합니다.
- **해결책**: 리트라이 루프 외부에 `raise last_exception` 구문을 반드시 추가합니다. 모든 폴백 경로가 유효한 `(ChatResponse, list[dict])` 튜플을 반환하는지 확인합니다.

### 2. 404 Model Not Found (로컬 에이전트)
- **증상**: `Agent: ollama (Model: llama3) - Error: 404 Not Found`.
- **원인**: 어댑터 기본값에 하드코딩된 모델명(`llama3` 등)이 실제 로컬 환경에 설치되어 있지 않음.
- **해결책**:
  - **런타임 탐색**: 실행 시점에 `adapter.discover_models()`를 호출합니다.
  - **제네릭 매핑**: 사용자가 보낸 제네릭한 힌트(`ollama`, `auto` 등)를 발견된 모델 리스트의 첫 번째 모델로 자동 매핑합니다.

### 3. 429 할당량 초과 (Gemini/Groq)
- **증상**: `Rate limit exceeded: 429`.
- **원인**: 무료 티어 사용량 제한 도달.
- **해결책**:
  - **엄격한 복구**: Probe 시도가 실제로 성공하기 전까지는 429 실패 키를 "복구됨"으로 표시하지 않습니다.
  - **제공자 폴백**: 특정 제공자의 모든 키가 429 상태면 즉시 다른 외부 제공자로 전환합니다.
  - **로컬 폴백**: 모든 외부 제공자가 실패하면 로컬 Ollama/OpenCode를 최후의 수단으로 사용합니다.

## 📈 로그 최적화 팁
- 모든 로그 엔트리에는 상관관계 추적을 위해 `[req_...]`가 반드시 포함되어야 합니다.
- `settings.debug` 활성화 시 전체 요청/응답 페이로드를 기록하여 데이터 변형을 추적합니다.
- 가독성을 위해 JSON 포맷을 지양하되, 스크립트 파싱이 가능하도록 메시지 구조를 일관되게 유지합니다.
